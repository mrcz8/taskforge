# Chapter 01 — Drizzle, the `users` Table, and the Migration Loop

**What this chapter covers:** how we added a real database layer on top of the Phase 0 scaffold — what Drizzle is and isn't, how drizzle-kit generates migrations from TypeScript, the Postgres driver choice, the programmatic migrator, and the file-by-file shape of everything we wrote. By the end of this chapter you should be able to add a new table, generate a migration, apply it, and reason about what happened in the database.

**What this chapter does NOT cover:** the GitHub Actions CI workflow, the Lucia auth layer, and factories/seeders. Those land in later chapters.

## 1. What Drizzle actually is

Drizzle is a **SQL query builder + schema toolkit** written in TypeScript. It is not a full ORM like Prisma or TypeORM.

| Tool | What it does | What it hides |
|---|---|---|
| Prisma | Own schema DSL (`schema.prisma`), generates a client, runs a background engine process | Hides the SQL almost entirely |
| TypeORM | Active-record / data-mapper, uses decorators | Hides joins behind "relations" |
| **Drizzle** | You write schema in TypeScript, query API reads like SQL (`db.select().from(users).where(...)`) | Hides almost nothing |

The upside: what you write looks like SQL, type-checks against your schema, and compiles to one function call — no hidden engine, no generated client, no magic. The downside: you have to know SQL. For a learning project, that's the feature.

### Two packages, two jobs

- **`drizzle-orm`** — the runtime. Imported by your app. Builds SQL and sends it through a driver.
- **`drizzle-kit`** — the CLI. Reads your TypeScript schema, diffs it against the last known snapshot, and writes migration SQL files.

They are deliberately separate. Your production bundle never ships `drizzle-kit`, which is why it lives under `devDependencies` and the runtime package lives under `dependencies`.

## 2. The driver question

Drizzle doesn't talk to Postgres itself. It hands SQL strings to a driver. In `drizzle-orm@0.36`, you have three realistic driver choices:

| Adapter | Driver package | Notes |
|---|---|---|
| `drizzle-orm/node-postgres` | `pg` | The old-guard Node Postgres client. Works everywhere. |
| `drizzle-orm/postgres-js` | `postgres` (by Porsager) | Modern, fast, promise-first. The community default for Drizzle. |
| `drizzle-orm/vercel-postgres` | `@vercel/postgres` | Serverless-specific, uses Vercel's edge Postgres gateway. |

TaskForge uses **`postgres-js`**. Reasons:

- It's the most widely documented Drizzle path, so error messages and examples on the internet match your stack.
- It uses **tagged template literals** natively (`sql\`SELECT ...\``), which is a nice fallback when you ever need raw SQL outside Drizzle.
- It runs under Bun without any adapter gymnastics.

> **Footnote on `Bun.sql`:** Bun 1.1+ ships a native Postgres driver as part of the runtime, and Drizzle has a `drizzle-orm/bun-sql` adapter for it — but only from `drizzle-orm@0.38+`. We're on 0.36 because that's what got installed when the project was scaffolded. Upgrading to get `Bun.sql` is a fine follow-up exercise but not required. The query API you write is identical either way — only the import and the driver constructor change.

## 3. The files we have on disk now

```
backend/
├── drizzle.config.ts          ← drizzle-kit's input
├── package.json                ← adds drizzle-orm, drizzle-kit, postgres
└── src/
    └── db/
        ├── index.ts            ← the `db` client your app imports
        ├── schema.ts           ← table definitions (source of truth)
        ├── migrate.ts          ← programmatic migrator
        └── migrations/
            ├── 0000_bright_phalanx.sql   ← generated by drizzle-kit
            └── meta/                      ← drizzle-kit's snapshots
```

Four files authored, one file (the SQL) generated. Let's go through each.

### 3a. `src/db/schema.ts` — the source of truth

```ts
import { pgTable, char, varchar, text, timestamp, uniqueIndex } from 'drizzle-orm/pg-core';

export const users = pgTable(
    'users',
    {
        id:           char('id', { length: 26 }).primaryKey(),
        email:        varchar('email', { length: 255 }).notNull(),
        name:         varchar('name', { length: 255 }).notNull(),
        passwordHash: text('password_hash').notNull(),
        createdAt:    timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),
        updatedAt:    timestamp('updated_at', { withTimezone: true }).defaultNow().notNull(),
    },
    (table) => ({
        emailUnique: uniqueIndex('users_email_unique').on(table.email),
    }),
);
```

Points worth absorbing:

- **`pgTable(name, columns, extras)`** — first argument is the physical table name, second is the column shape, third is for table-level constraints (indexes, composite uniques, check constraints).
- **`char('id', { length: 26 })`** — `CLAUDE.md` mandates ULIDs as `char(26)`. UUIDs are 36 characters; a ULID is a 26-character Crockford-base-32 encoded 128-bit value. Why ULID instead of UUID: ULIDs are **lexicographically sortable by creation time** — `ORDER BY id ASC` gives you chronological order for free, which is great for pagination.
- **`timestamp(..., { withTimezone: true })`** compiles to Postgres `timestamp with time zone` (aka `timestamptz`). `CLAUDE.md` requires this: "Timestamps are `timestamptz`, stored UTC."
- **`.defaultNow().notNull()`** — chained builder calls. Order does not matter functionally.
- **No `id: serial()`.** Auto-increment integers are forbidden here because IDs leak row order and create merge conflicts across workspaces later.
- **`(table) => ({ emailUnique: ... })`** — the **third argument to `pgTable`** is a function returning an object of extras. Drizzle passes the column map in so you can reference `table.email`. The keys of the returned object are the TypeScript-side names; the SQL index name is the string you pass to `uniqueIndex(...)`.
- **Index naming convention** — `users_email_unique` prefixes the table name and ends with the kind. This makes `psql \di` output readable at a glance. A generic `email_index` would be ambiguous across tables.

#### A subtle TS detail: implicit-any on `table`

The original attempt wrote `(table) => ({...})` and hit a TS diagnostic: "Parameter 'table' implicitly has an 'any' type". That only shows up until TypeScript can actually resolve `drizzle-orm/pg-core` — once the types are available (editor-side `bun install` populating host `node_modules`), `pgTable` provides the correct callback type and `table` becomes strongly typed without any annotation. In other words: the diagnostic was a symptom of the editor not seeing the types, not of bad code.

### 3b. `drizzle.config.ts` — drizzle-kit's instructions

```ts
import { defineConfig } from 'drizzle-kit';

export default defineConfig({
    dialect: 'postgresql',
    schema: './src/db/schema.ts',
    out: './src/db/migrations',
    dbCredentials: {
        url: process.env.DATABASE_URL!,
    },
    strict: true,
    verbose: true,
});
```

**The four fields you'll see in every Drizzle config:**

- **`dialect`** — the SQL flavor. Valid values in drizzle-kit are exactly `'postgresql' | 'mysql' | 'sqlite' | 'turso'`. The string `'postgres'` is rejected — a surprisingly common typo that produces a ZodError with the four valid options listed.
- **`schema`** — path to your schema file (or a glob). Paths are relative to the config file, which lives at `backend/drizzle.config.ts`, so `./src/db/schema.ts` is correct. `../src/db/schema.ts` would walk *up* to the project root and fail.
- **`out`** — where generated SQL lands. Same relative-to-config rule. `./src/db/migrations` is a directory drizzle-kit creates on first run.
- **`dbCredentials.url`** — used by `drizzle-kit migrate` and `drizzle-kit studio`. The migrator we actually run in the app (`migrate.ts`, see below) reads `DATABASE_URL` directly, so this field only powers the CLI commands.

`strict: true` and `verbose: true` make drizzle-kit louder — it'll print the statements it's about to execute. For a learning project that's what you want.

### 3c. `src/db/index.ts` — the runtime client

```ts
import { drizzle } from 'drizzle-orm/postgres-js';
import postgres from 'postgres';
import * as schema from './schema';

const connectionString = process.env.DATABASE_URL;
if (!connectionString) {
    throw new Error('DATABASE_URL is not set');
}

const client = postgres(connectionString);

export const db = drizzle(client, { schema });
export type DB = typeof db;
```

Unpacking:

- **`postgres(connectionString)`** — constructs a connection *pool*, not a single connection. The `postgres` package manages connections lazily: a pool is opened on first query, sized to a default of 10, and reused across queries. You do not manually open/close connections in application code.
- **`drizzle(client, { schema })`** — wraps the driver so Drizzle can send SQL through it. Passing `{ schema }` is what unlocks the **relational query API** (`db.query.users.findMany(...)`), which is Drizzle's higher-level, JOIN-aware interface. Without `{ schema }`, you're limited to the lower-level builder (`db.select().from(users)...`). Both work; the relational API is nicer for nested reads.
- **`export type DB = typeof db`** — exporting the type of the `db` singleton lets tests and actions declare parameters as `db: DB` without having to reconstruct the inferred type. Drizzle is types-first; you will lean on `typeof` a lot.
- **One `db` per process.** Drizzle and the driver both pool internally — do not construct multiple `db` instances.

### 3d. `src/db/migrate.ts` — the programmatic migrator

```ts
import { drizzle } from 'drizzle-orm/postgres-js';
import { migrate } from 'drizzle-orm/postgres-js/migrator';
import postgres from 'postgres';

const connectionString = process.env.DATABASE_URL;
if (!connectionString) {
    throw new Error('DATABASE_URL is not set');
}

const migrationClient = postgres(connectionString, { max: 1 });

await migrate(drizzle(migrationClient), {
    migrationsFolder: './src/db/migrations',
});

await migrationClient.end();
```

Why this file exists at all, instead of using `drizzle-kit migrate` from the CLI:

- **`drizzle-kit migrate`** is a perfectly fine command — it reads `drizzle.config.ts`, opens its own connection, runs unapplied migrations, and exits.
- **The programmatic migrator** runs through a connection *your app* opens. That means it uses the same env vars, the same driver configuration, the same process. In production, you typically run this as a one-off before the app boots; having it as a file in your repo means "migrate on deploy" is just `bun run src/db/migrate.ts`.

Line-by-line notes:

- **`postgres(connectionString, { max: 1 })`** — one connection is enough for a migration run, and deliberately capping the pool avoids accidentally hoarding connections when you're running on a constrained Postgres instance.
- **Top-level `await`** works because Bun runs ES modules natively and Bun supports top-level await. In a Node ESM project this works too; in a CommonJS project you'd need an async IIFE.
- **`await migrationClient.end()`** — explicitly closes the pool so the Node/Bun process exits. Without it, the script hangs waiting for idle connections to time out.

### 3e. The generated SQL: `0000_bright_phalanx.sql`

The filename has two parts: a four-digit index (`0000`) and a random two-word suffix (`bright_phalanx`). The index is what Drizzle sorts migrations by; the suffix is just to make filenames humane. You can pass `--name my_name` to `drizzle-kit generate` to control the suffix.

Contents:

```sql
CREATE TABLE IF NOT EXISTS "users" (
    "id" char(26) PRIMARY KEY NOT NULL,
    "email" varchar(255) NOT NULL,
    "name" varchar(255) NOT NULL,
    "password_hash" text NOT NULL,
    "created_at" timestamp with time zone DEFAULT now() NOT NULL,
    "updated_at" timestamp with time zone DEFAULT now() NOT NULL
);
--> statement-breakpoint
CREATE UNIQUE INDEX IF NOT EXISTS "users_email_unique" ON "users" USING btree ("email");
```

Notes on the generated output:

- **`IF NOT EXISTS`** on both the table and the index — drizzle-kit emits idempotent-ish SQL, so if two teammates happen to run the same migration against the same database at nearly the same time, the second run fails cleanly rather than corrupts.
- **`--> statement-breakpoint`** is a **marker**, not SQL. The migrator splits the file on these to run statements one at a time. Some Postgres statements (like `CREATE INDEX CONCURRENTLY`) refuse to run inside a transaction; the breakpoints let each one execute in its own boundary.
- **`USING btree`** — explicit index type. B-tree is the default in Postgres, but drizzle-kit emits it anyway for clarity.
- **Column quoting** — all identifiers are double-quoted. This makes them case-sensitive in SQL, which matters if you ever read this by hand.

### 3f. The migration bookkeeping table

Applying a migration writes a row to `drizzle.__drizzle_migrations`:

```
 id |                               hash                               |  created_at
----+------------------------------------------------------------------+---------------
  1 | 34d6aa60e26276f52ef94515ac2862de2af0ffc25d73a8a7ecb267dbee618d01 | 1776840256125
```

Three things to notice:

- **The table lives in a dedicated `drizzle` schema**, not in `public`. This keeps it out of `pg_dump --schema=public` and out of your relational query surface.
- **`hash`** is the SHA-256 of the migration file's contents, not its filename. If you edit an already-applied migration, Drizzle will notice the hash mismatch at the next `migrate()` call and error. This is a deliberate safety net — edited applied migrations are almost always a mistake.
- **`created_at` is a `bigint` of Unix milliseconds**, not a `timestamptz`. Quirky but internal; you'll probably never query it.

## 4. The workflow you just learned

Every schema change from now on will follow this loop:

1. **Edit `schema.ts`** — add a column, add a table, add an index.
2. **Generate the SQL** — `docker compose exec backend bun run db:generate`. Drizzle-kit diffs your schema against `migrations/meta/` and emits a new `NNNN_*.sql` file.
3. **Read the SQL before applying.** Always. This is where you catch things like "I meant `text` but typed `varchar`" or "I forgot to add the index I thought I added."
4. **Apply it** — `docker compose exec backend bun run db:migrate`. Silent success, failure with a reason.
5. **Verify with `psql`** — `docker compose exec postgres psql -U taskforge -d taskforge -c '\d <table>'`.
6. **Commit the generated SQL and the meta files together.** Schema-as-code only works if the SQL and the snapshots in `meta/` travel together.

## 5. Two rough edges worth remembering

Two issues we tripped over that are not really Drizzle's fault but will happen again.

### 5a. EACCES writing into `node_modules`

The backend container runs as `USER bun` (non-root), but the image's `node_modules` was copied in as root, so the named `backend_node_modules` volume was initialized with root-owned files. Running `bun install` as `bun` gives `EACCES`.

**Host-side `sudo` doesn't help.** It elevates the Docker CLI on your host, but the container's `USER` is fixed.

The two-line fix:

```bash
docker compose exec -u root backend bun install
docker compose exec -u root backend chown -R bun:bun node_modules
```

The permanent fix is a `--chown=bun:bun` on the `COPY` lines in `backend/Dockerfile`. That change belongs in its own small cleanup chapter.

### 5b. The IDE shows "Cannot find module" everywhere

Your IDE's TypeScript server runs on your host; the container's `node_modules` is behind a volume mount the host can't see. The code runs correctly — it's only the editor that's confused.

Simplest fix: `cd backend && bun install` on the host as well. Docker ignores the host `node_modules` because the named volume layers over it inside the container. You end up with two copies of dependencies, which is the price of this setup.
